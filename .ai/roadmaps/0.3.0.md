# Roadmap 0.3.0

`0.3.0` turns the released memory MVP into a self-hosted, multi-tenant SaaS-style
memory service that supports coding agents and human agents with strict isolation,
with workspaces (not tenants) as the operator's day-to-day mental model.

This release codifies three decisions made in earlier sessions:

1. **Workspace is not a tenant.** A workspace is a directory; a tenant is a
   universe of data. The relationship between them is many-to-many with
   explicit `tenant_access` rows.
2. **Human agents require Postgres.** Coding agents ask the user. Self-hosted
   servers use Postgres. SQLite is the local-first default.
3. **alfred init prompts safety guards** when a workspace conflicts with
   another in the same directory tree.

## Phase 13: alfred_registry

`packages/memory-server/migrations/000_alfred_registry.sql` with the
control-plane SQLite. Two TRIGGERs:

- `prevent_tenant_delete_with_readers` — block cascade orphans.
- `check_no_dual_postgres_in_hierarchy` — refuse two distinct Postgres
  tenants in a workspace chain.

## Phase 14: TenantService

TDD the tenant service with:

- `provision` (validates hosting policy invariants).
- `delete` (cascades to DB file or DB drop).
- `findAncestors` and `findDescendants` for the init flow's safety scan.

## Phase 15: UserService

TDD the user service with scrypt-based key hashing (configurable algorithm
column), rotation, revocation, and `WHERE user_id = $1` discipline.

## Phase 16: Sessions, Topics, ACs

The Session / Topic / AcceptanceCriterion services with state machines and
derived rollups. Migration `004_session_topic_ac.sql`.

## Phase 17: CLI

`alfred init`, `alfred provision`, `alfred list`, `alfred key rotate`,
`alfred key revoke`, `alfred key list`, `alfred migrate`,
`alfred validate-policy`, `alfred tenant change-kind`. Interactive prompts
via `@clack/prompts` with TDD.

## Phase 18: Server

`alfred-memory-server.mjs` bin with `ALFRED_MEMORY_HOSTING=local|self-hosted`
mode switch. Tenant router resolves backend per tenant, opens a `pg.Pool`
or `better-sqlite3` connection accordingly.

## Phase 19: SQLite stores

`sqlite-user-store`, `sqlite-tenant-store`, `sqlite-session-store`,
`sqlite-topic-store`, `sqlite-ac-store`, `sqlite-memory-store`. WAL mode,
`busy_timeout = 5000`, feature parity with Postgres.

## Phase 20: Cross-tenant and policy tests

15+ vectors of intentional cross-tenant leak attempts. Plus negative tests
for the two new policies (no SQLite for human agents, no dual-Postgres in
hierarchy).

## Constraints

- Local-first for **compute** (memory persistence can be local or remote).
- Provider calls allowed: `0`.
- Core remains harness-agnostic.
- Permission escalation and harness config writes require human approval.
- Baseline updates require human approval.
- Human agents require Postgres.
- Coding agents ask the user on `alfred init`.
- Workspace invariant: one tenant at init time.
- Tenant isolation: one physical DB per tenant.

## Out of Scope (deferred)

- Native Anthropic / Google Gemini adapters (v0.3.1+).
- Federated sync (v0.4).
- Hybrid offline cache (v0.4).
- Multi-agent collaboration sessions.
- Embedding-based semantic search.
