# Roadmap 0.4.0

`0.4.0` turns the SaaS Web Onboarding from v0.3.1 into a production-ready
service with anti-bot protection, email verification, real-database
isolation tests in CI, key recovery, and local embedding-based semantic
search. Zero provider calls in any code path.

This release codifies four decisions from earlier sessions:

1. **CAPTCHA is optional, not required.** When the operator sets
   `ALFRED_TURNSTILE_SITE_KEY` + `ALFRED_TURNSTILE_SECRET_KEY`, the
   server requires and verifies a Turnstile token. When unset, the
   endpoint behaves exactly as in v0.3.1.
2. **Email is optional too.** SMTP is configured via standard env vars
   (`ALFRED_SMTP_*`). If unconfigured, the server logs a warning and
   skips email sending. Local dev and CI run without SMTP.
3. **The embedding model runs locally.** We use
   `@xenova/transformers` with `Xenova/all-MiniLM-L6-v2` (22MB, MIT).
   No OpenAI, no Anthropic, no external API. The model is downloaded
   once on first use and cached at `~/.cache/transformers/`.
4. **The model is a fallback, not a hard dependency.** If the model
   fails to load (offline, missing dep), `searchMemories` falls back
   to keyword search with a logged warning.

## Phase 25: CAPTCHA (Cloudflare Turnstile)

- New `packages/memory-server/src/bootstrap/captcha-verifier.js`
  wrapping the `siteverify` endpoint at
  `https://challenges.cloudflare.com/turnstile/v0/siteverify`.
- `POST /console/api/bootstrap` accepts `X-Turnstile-Token` header
  (and `turnstile_token` in body as a fallback for cross-origin
  clients without custom-header support).
- The web signup form includes the Turnstile widget when
  `window.ALFRED_TURNSTILE_SITE_KEY` is set in the injected build
  config; otherwise the form skips the widget.
- Tests: valid token, invalid token, expired token, missing config.

## Phase 26: Email Verification

- New `tenant_email_verifications` table:
  `id, tenant_id, email, token, expires_at, used_at, created_at`.
- `POST /console/api/bootstrap` accepts optional `email` field.
  When provided, server generates a 32-char random token, stores it,
  and (if SMTP is configured) sends a magic link.
- New `GET /console/api/verify?token=...` endpoint that marks the
  token as used and, if not already issued, emits the API key.
- `email-sender.js` is a thin Nodemailer wrapper. If
  `ALFRED_SMTP_HOST` is unset, the sender is a no-op that logs the
  would-be email.
- Tests: token valid, expired, reused, SMTP off.

## Phase 27: CI Postgres Isolation Tests

- New `.github/workflows/ci-postgres.yml` with a `postgres:16-alpine`
  service and a job that runs
  `packages/memory-server/test/cross-tenant-isolation-postgres.test.mjs`
  against a real cluster.
- The test creates two `server_managed` tenants via
  `createBootstrap` and asserts that one tenant's queries return
  zero rows from the other's tables — using the production code
  path, not a mock.
- The gate blocks PRs that fail this test.
- Tests: 15 existing + 1 new = 16 vectors.

## Phase 28: Forgot-My-Key

- New `tenant_recoveries` table:
  `id, tenant_id, email, token, expires_at, used_at, created_at`.
- `POST /console/api/recover` with `{ email }` body, rate-limited to
  3 attempts per IP per hour. Server generates a 1-hour token, sends
  a magic link with the new key, marks the old key as revoked.
- `GET /console/api/recover?token=...` validates, issues new key,
  revokes old.
- Web panel in the locked view: "Forgot your key? Enter your email
  and we'll send a recovery link."
- Tests: token valid, expired, reused, rate limit, key rotation.

## Phase 29: Embedding-Based Semantic Search

- New `memory_embeddings` table:
  `memory_id, model, embedding BLOB, created_at`. Embedding is a
  Float32Array of 384 dimensions serialized as a BLOB.
- `embedder.js` lazy-loads `@xenova/transformers` and the
  `Xenova/all-MiniLM-L6-v2` model. Downloads the model on first
  use; subsequent uses hit the local cache.
- `semantic-index.js` computes cosine similarity between the query
  embedding and all stored embeddings, then ranks by score.
- `searchMemories` accepts a `mode` option: `semantic`, `keyword`,
  or `hybrid` (default). Hybrid = RRF (Reciprocal Rank Fusion) of
  semantic and keyword rankings.
- Background worker: when a memory is created or updated, enqueue
  an embedding job. The job is best-effort; if the model is
  unavailable, the memory is stored without an embedding and only
  keyword search will find it.
- Tests: cosine similarity, hybrid ranking, model unavailable
  (fallback to keyword), no-results.

## Constraints

- Local-first for **compute** (memory persistence can be local or remote).
- Provider calls allowed: `0`.
- Core remains harness-agnostic.
- Permission escalation and harness config writes require human approval.
- Baseline updates require human approval.
- CAPTCHA, email, and the embedding model are all optional — the
  server runs with a minimal config (no CAPTCHA, no SMTP, no model)
  and gracefully degrades.
- The embedding model is downloaded at boot, not at signup. The
  first search after a fresh install will be slow; subsequent
  searches are fast.

## Out of Scope (deferred to v0.5.0)

- Federated sync between agent instances.
- Hybrid offline cache.
- Multi-agent collaboration sessions.
